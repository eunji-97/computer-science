# 아키텍처(Architecture)
시스템의 기본 뼈대와 구조, 구성 요소 간 상호 작용 방식을 정의하는 것이다. (설계와는 다르다.)

각 개발 과정의 설계 결정을 통제하고 전반적인 틀을 제시, 이를 이용해 각 이해 관계자들의 의사소통을 돕는다.

---

## 원칙
1. 관심사 분리
	소프트웨어가 수행하는 작업의 종류에 따라 분리되어야 한다.
	분리된 모듈은 각 역할이 명확해야 하며, 정의된 문제를 해결할 수 있게 구성되어야 한다.
2. 느슨한 결합
	모듈 간 의존성을 낮춰 각 모듈이 독립적으로 동작할 수 있도록 한다. 한 구성 요소의 변경이
	다른 구성 요소에 최소한의 영향만 미치도록 설계하는 것을 의미한다.
3. 높은 응집력
	하나의 모듈 내부 요소들이 서로 밀접하게 관련된 기능을 수행하도록 구성되는 것을 의미한다.
	하나의 모듈은 하나의 명확한 책임을 가져야 한다.
4. 확장성
	소프트웨어는 확장 가능해야 한다.  방법엔 두가지가 있는데,
		1. 수직적 확장 : 단일서버에 더 많은 cpu, 메모리와 같은 컴퓨팅 성능을 추가
		2. 수평적 확장 : 시스템에 더 많은 서버를 추가

좋은 아키텍처는 소프트웨어에 품질 확보, 유연함과 유지보수에 용이하다는 이점을 준다.
